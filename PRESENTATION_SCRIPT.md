## Frame 1 — Title / Setup

**Visual:** Two side-by-side cards. Left: "Pyramidal tossup" with bullet points about clue difficulty. Right: "Multiple-choice twist" showing a 2×2 answer grid (A/B/C/D).

**Script:**

Quiz bowl is a pyramidal trivia format: questions are read clue by clue, starting obscure and getting easier. Our twist is converting this into a **multiple-choice** setting with four fixed options visible from the start.

Three key bullets on the left:
- Obscure clue — only experts get it.
- Stronger clue — narrows the field.
- Giveaway clue — makes the answer obvious.

The **RL intuition** is: the answer space is finite; the hard part is deciding *when* to stop listening. At each prefix the policy asks: should I **WAIT** or **BUZZ** with one of the answers?

**Script:**

Now that we've seen the posterior sharpen, let's formalize the decision. At each step the agent compares two values: the value of committing now vs. the value of waiting for one more clue.

## Frame 14 — Section Header: Abstain Semantics

**Visual:** Centered section card with red accent. "Section 3: abstain means continue, not NOTA."

**Script:**

A crucial semantic point that trips people up. When the model *abstains*, it does **not** mean "none of the above." Let's clarify.

---

## Frame 15 — Abstain Stage 0

**Visual:** Left card: 2×2 answer grid reminding that the gold answer is already in the option set. Right card: nearly uniform bars (30/24/24/22). "ABSTAIN" label. Red X over "NOTA".

**Script:**

The gold answer is always in the option set. When the model outputs "abstain," it means: *I'd rather hear one more clue than commit right now.* It's a **continuation** action, not a rejection of the answer choices.

This is important because: only architecture or loss design artifacts can make abstain behave like a reject-option sink. That's an optimization artifact, not the Bayes-optimal semantics.

---

## Frame 16 — Abstain Stage 1

**Visual:** Same layout, bars now show A at 52%. Still "ABSTAIN".

**Script:**

Even when the correct answer is pulling ahead, waiting can still be rational. The gold answer is present *and* plausible, but another clue has positive value of information. Abstain = "the marginal clue is worth it."

---

## Frame 17 — Abstain Stage 2

**Visual:** Bars show A at 78%. Still labeled "ABSTAIN" conceptually, but nearing the threshold.

**Script:**

At this point abstain is about continuation value, not exhaustivity of the answer set. The takeaway: **abstain ≠ NOTA.** It means the expected gain from one more clue outweighs the expected gain from committing now.

---

## Frame 18 — Section Header: Factorization

**Visual:** Centered section card with purple accent. "Section 4: factor answer quality and stop timing."

**Script:**

Now the implementation insight. Instead of training one monolithic policy over 5 actions (WAIT + 4 answers), we can **factor** the decision into two modules.

---

## Frame 19 — Factorization Stage 0

**Visual:** Flow diagram. Left: "prefix h_t + option set C" feeds into two modules. Top module (blue): "Answer module → outputs p_ans(i)." Bottom (orange, dimmed): "Stop module." Right: "Optional flat interface" card.

**Script:**

First, the **answer module**: given the current prefix and options, it outputs a distribution over answers — p_ans(i | h_t). This module focuses purely on *which* answer is best, ignoring the when-to-stop question.

---

## Frame 20 — Factorization Stage 1

**Visual:** Same diagram, now the stop module is highlighted orange. The flat interface card on the right shows: P(WAIT) = 1 − p_buzz, P(BUZZ i) = p_buzz × p_ans(i).

**Script:**

Second, the **stop module**: it outputs p_buzz(h_t), a scalar probability of buzzing at this timestep. The two modules compose cleanly:

- P(WAIT) = 1 − p_buzz
- P(BUZZ on answer i) = p_buzz × p_ans(i)

This gives the same decision semantics as a flat 5-action policy, but with a much cleaner implementation. The RL agent then only needs to learn the *stopping* problem.

---

## Frame 21 — Section Header: Practical Recipe

**Visual:** Centered section card with green accent. "Section 5: one clean mental model."

**Script:**

Let's put it all together into a concrete training recipe.

---

## Frame 22 — Training Recipe

**Visual:** Three colored cards in a row: (1) blue — "answer model first: learn p_ans(i)"; (2) orange — "hazard-loss warm start: learn when to commit"; (3) green — "PPO fine-tuning: optimize task reward."

**Script:**

The three-stage pipeline:

1. **Answer model first** — train the answer distribution with supervised learning. This is a standard MCQA head.
2. **Hazard-loss warm start** — train the stop module with a survival-analysis-style loss so it learns a reasonable baseline for when to commit.
3. **PPO fine-tuning** — let RL optimize the full task reward (correct answer + timing bonus). Because the answer module is already good, RL focuses on the stopping decision.

The key insight: RL learns *mostly* the stopping problem, not which-answer-plus-whether-to-stop all at once.
